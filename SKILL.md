# Генератор учебных отчётов ИТК

Навык собирает DOCX-отчёт по учебной практике колледжа так, чтобы он
прошёл проверку: правильная структура, шрифты, поля, **рамка и штампы
скопированы из шаблона-эталона** ученика, а не нарисованы программно.

<core_principle>
ГЛАВНЫЙ ПРИНЦИП: рамки и штампы НЕ создавать с нуля. Они живут как
сложные VML/DrawingML фигуры в `header1.xml` и в `<wp:anchor>` первого
параграфа тела документа-эталона. Любая попытка собрать их таблицами
через python-docx даёт визуально неточный результат и приводит к
переделке. Бери шаблон ученика, подменяй текст, пересобирай только тело.
</core_principle>

## Когда активировать

Срабатывай, если пользователь:

- просит «сделай отчёт», «собери отчёт по учебной практике», «отчёт по
  ИТК», «по колледжу», «учебный отчёт для Edupage»;
- упоминает «рамка», «штамп», «большой штамп», «малый штамп»,
  «колонтитулы», «ГОСТ-оформление DOCX»;
- задаёт тему из списка «Базы данных» (BI, NoSQL, MongoDB, Redis,
  Hadoop, ETL, миграция, мониторинг, оптимизация запросов и т.п.);
- даёт DOCX-шаблон ученика и просит «сделай по этому образцу».

## Высокоуровневый сценарий

<workflow>
1. **Собери входные данные.** Получи от пользователя:
   - тему отчёта;
   - принятый отчёт-эталон в виде `.docx` (без него — стоп, проси);
   - ФИО студента, курс, группу, специальность, квалификацию, даты
     практики, ФИО руководителя;
   - подтемы для специальной части (при необходимости);
   - программу для скриншотов (если Power BI/Tableau не запускается
     под Linux — предложи Metabase/Superset).

2. **Подготовь окружение.** ВСЕГДА работай в venv:
   ```bash
   python3 -m venv ~/report_env
   ~/report_env/bin/python -m pip install python-docx lxml Pillow
   ```
   Конвертация в PDF: `libreoffice --headless --convert-to pdf …`.

3. **Открой `references/report_rules.md`** и прочитай разделы
   `<template_strategy>`, `<text_substitution>`, `<body_rebuild>`,
   `<formatting_rules>`, `<figures_and_tables>`, `<anti_patterns>`.

4. **Возьми шаблон ученика как основу** (`shutil.copy`).
   Подмени в нём только тексты (ФИО, тема, группа, курс, даты,
   число листов). НЕ ТРОГАЙ структуру `<w:sectPr>`, `header1.xml`,
   `header2.xml`, `<wp:anchor>` с большим штампом.

5. **Удали старое тело** между якорным параграфом (с `<wp:anchor>`) и
   финальной `<w:sectPr>`.

6. **Сгенерируй своё тело** в порядке:
   - СОДЕРЖАНИЕ (с реальными номерами страниц)
   - ВВЕДЕНИЕ
   - 1 ОСНОВНАЯ ЧАСТЬ → 1.1, 1.2, … (с рисунками и подписями)
   - 2 СПЕЦИАЛЬНАЯ ЧАСТЬ → 2.1, 2.2, …
   - ЗАКЛЮЧЕНИЕ
   - СПИСОК ИСПОЛЬЗОВАННОЙ ЛИТЕРАТУРЫ (≥15 источников)

7. **Сохрани DOCX** и сконвертируй в PDF.

8. **Проверь по чек-листу ниже** и только потом отдавай файл.

9. **Отдай пользователю** оба файла (`.docx` и `.pdf`) с латинскими
   именами (платформа не любит кириллицу в attachments). Предупреди:
   рамка и большой штамп в просмотрщике PDF от LibreOffice могут не
   отрисоваться — это нормально, в Word всё ок.
</workflow>

## Где искать что

<file_map>
- `references/report_rules.md` — полные правила: окружение, формат,
  стратегия шаблона, подмена текстов, перестройка тела, рисунки и
  таблицы, библиография, антишаблоны. Читай ПЕРЕД генерацией.
- `assets/report_template.md` — структурный трафарет готового отчёта:
  заголовки, абзацы-плейсхолдеры, места под рисунки и таблицы. Используй
  как визуальный шаблон при заполнении тела.
- `assets/build_from_student.py` — рабочий скрипт-скелет (Python). Уже
  умеет копировать шаблон, подменять тексты, удалять старое тело и
  собирать новое.
</file_map>

## Чек-лист перед выдачей

<validation_checklist>
- [ ] Файл открывается в `Document(str(path))` без исключений
- [ ] Структура: титул → СОДЕРЖАНИЕ → ВВЕДЕНИЕ → 1 ОСНОВНАЯ ЧАСТЬ →
      2 СПЕЦИАЛЬНАЯ ЧАСТЬ → ЗАКЛЮЧЕНИЕ → СПИСОК (опционально ПРИЛОЖЕНИЯ)
- [ ] Шрифт: Times New Roman 14 в основном тексте, 12 в подписях
- [ ] Жирного текста **нет** нигде в теле
- [ ] Курсив только в подписях рисунков
- [ ] Поля 30/10/15/30 мм (`<w:pgMar>` в финальной `<w:sectPr>`)
- [ ] Межстрочный 1.0, абзацный отступ 1.25 см
- [ ] ≥ 27 страниц (проверь `pdfinfo file.pdf | grep Pages`)
- [ ] СОДЕРЖАНИЕ соответствует фактическим номерам страниц
- [ ] Каждый рисунок имеет подпись «Рисунок-N.M Описание» и ссылается
      в тексте `(рисунок N.M)`
- [ ] Каждая таблица имеет подпись «Таблица-N.M Описание» сверху
- [ ] Источников в списке литературы **≥ 15**, формат с указанием
      даты обращения для электронных
- [ ] `diff` `header1.xml` с шаблоном-эталоном пуст — рамка и малый
      штамп скопированы байт-в-байт
- [ ] `<wp:anchor>` с большим штампом в первом параграфе тела — на
      месте, с подменённым ФИО/группой/темой
</validation_checklist>

## Запрещённые действия

<anti_patterns>
- ❌ Создавать штампы программно через `python-docx` Table API
- ❌ Добавлять `<w:pgBorders>` для имитации рамки
- ❌ Менять `<w:titlePg/>` и `<w:headerReference>` в `<w:sectPr>` шаблона
- ❌ Удалять `<wp:anchor>` из первого параграфа секции тела
- ❌ Использовать `<w:b/>` (жирный) в теле документа
- ❌ Делать сквозную нумерацию рисунков «Рисунок 1, Рисунок 2…»
      (только «Рисунок-1.1», «Рисунок-1.2»)
- ❌ Ставить точку после номера подраздела («1.1.»)
- ❌ Менять стили титульного листа (жирность, размер, выравнивание)
- ❌ Запускать `pip install` в системный python (только venv)
- ❌ Класть служебные файлы (планы, todo, скрипты-черновики) в финальную
      выдачу пользователю
- ❌ Генерировать отчёт **без шаблона ученика** — попроси шаблон
</anti_patterns>
