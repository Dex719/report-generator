# Полные правила генерации учебных DOCX-отчётов ИТК (ГОСТ)

Этот файл — справочник для навыка `report-generator`. Прочитай нужные
разделы перед сборкой отчёта. Все правила проверены на практике, после
нескольких итераций. Главный принцип: **рамки и штампы — из шаблона,
тексты — подменяй, тело — пересобирай**.

---

## 1. Окружение и инструменты

<environment>

</environment>

## 2. Обязательные входные данные

<required_inputs>
Перед генерацией собери у пользователя:

1. **Тему отчёта** (например: «Визуализация данных. Использование
   инструментов BI (Power BI, Tableau)»).
2. **Шаблон ученика** — принятый отчёт в формате `.docx`. Без него
   стоп: попроси шаблон. В нём уже есть:
   - правильный титульный лист;
   - section break'и;
   - `word/header1.xml` (рамка + малый штамп);
   - `word/header2.xml` (обычно пустой, для первой страницы секции);
   - `<wp:anchor>` с большим штампом в первом параграфе тела.
3. **Личные данные**: ФИО студента, курс, группа, специальность,
   квалификация, период практики, ФИО руководителя.
4. **Подтемы для специальной части** (если у отчёта есть «2 СПЕЦИАЛЬНАЯ
   ЧАСТЬ»).
5. **Инструмент для скриншотов**. Если требуется живой BI (Power BI,
   Tableau) — предупреди, что нативно под Linux они не запускаются.
   Предложи Metabase или Apache Superset (open-source, ставятся одной
   командой).
</required_inputs>

## 3. Общая структура отчёта

<structure_requirements>
Порядок разделов (каждый — с новой страницы):

```
Титульный лист                  (1 страница, без рамки и штампа)
СОДЕРЖАНИЕ                      (центр, прописными; рамка + БОЛЬШОЙ штамп)
ВВЕДЕНИЕ                        (центр, прописными)
1 ОСНОВНАЯ ЧАСТЬ                (с абзацного отступа, прописными)
   1.1 …
   1.2 …
   1.3 …
   1.4 …
   1.5 …
2 СПЕЦИАЛЬНАЯ ЧАСТЬ             (с абзацного отступа, прописными)
   2.1 …
   2.2 …
   2.3 …
   2.4 …
ЗАКЛЮЧЕНИЕ                      (центр, прописными)
СПИСОК ИСПОЛЬЗОВАННОЙ ЛИТЕРАТУРЫ (центр, прописными; ≥ 15 источников)
ПРИЛОЖЕНИЯ                      (только если попросил пользователь)
```

- Минимальный объём — **27 страниц** (без приложений). Регламент пишет
  25, но пользователь требует 27.
- «Структурные» заголовки (СОДЕРЖАНИЕ, ВВЕДЕНИЕ, ЗАКЛЮЧЕНИЕ, СПИСОК) —
  по центру, прописными, не нумеруются.
- «Разделы» (1 ОСНОВНАЯ ЧАСТЬ, 2 СПЕЦИАЛЬНАЯ ЧАСТЬ) — с абзацного
  отступа, прописными, нумерация арабская.
- Подразделы (1.1, 1.2, 2.1, …) — с абзацного отступа, с прописной
  буквы, **без точки** после номера и в конце.
- Каждый структурный элемент и каждый раздел начинаются **с новой
  страницы** (`<w:pageBreakBefore/>` или `<w:br w:type="page"/>`).
</structure_requirements>

## 4. Форматирование текста

<formatting_rules>
| Параметр                | Значение                                       |
|-------------------------|------------------------------------------------|
| Шрифт основного текста  | Times New Roman, 14 pt                         |
| Шрифт подписей рисунков | Times New Roman, 12 pt, **курсив**, по центру   |
| Шрифт подписей таблиц   | Times New Roman, 12 pt, прямой, с абзацного отступа |
| Межстрочный интервал    | 1.0 (`line=240`, `lineRule=auto`)              |
| Абзацный отступ         | 1.25 см (≈567 twips на см)                     |
| Выравнивание основного  | По ширине (`jc=both`)                          |
| Поля                    | Левое 30 мм, правое 10 мм, верх 15 мм, низ 30 мм |
| Размер страницы         | A4 (210 × 297 мм)                              |
| Нумерация страниц       | Арабские, правый нижний угол, **внутри малого штампа** |
| Титульный лист          | Считается в нумерации, но номер не печатается  |
| Цвет текста             | Чёрный (#000000)                               |

<bold_rule>
**Жирный текст (`<w:b/>`) в теле документа категорически запрещён.**
Допустим только если он унаследован от стилей титульного листа из
шаблона. Не добавляй `bold=True` ни в один Run, который пишешь сам.
</bold_rule>

<italic_rule>
Курсив (`<w:i/>`) — только в подписях рисунков. Больше нигде.
</italic_rule>
</formatting_rules>

## 5. КЛЮЧЕВАЯ СТРАТЕГИЯ: шаблон-как-основа

<template_strategy>

### 5.1 Почему так

Раньше я создавал колонтитулы и штампы программно: строил таблицы
через `python-docx`, добавлял `<w:pgBorders>` для рамок, рисовал
штампы Table API. Это работало частично — таблица собиралась, но
визуально отличалась от принятых отчётов. Пользователь прямо сказал:

> «Возьми отсюда колонтитулы и используй их в моем отчете. Нельзя их
> самому воссоздовать а нужно их отсюда скопировать».

Шаблон ученика — это уже скомпонованный документ Word, в котором
рамка и штампы вынесены в XML-структуры:

- `word/header1.xml` — большая сгруппированная VML-фигура: рамка по
  периметру содержательной части + малый штамп с полем `PAGE` для
  номера страницы. Используется на всех страницах кроме первой
  страницы секции.
- `word/header2.xml` — пустой (или почти пустой). Используется на
  «первой странице» секции (когда у секции включён `<w:titlePg/>`).
- В теле документа (`word/document.xml`), в **первом параграфе секции
  с СОДЕРЖАНИЕм**, лежит `<wp:anchor>` со ещё одной сгруппированной
  фигурой: рамка + БОЛЬШОЙ штамп (для страницы СОДЕРЖАНИЕ).

Эти структуры — десятки килобайт XML, которые невозможно «угадать»
из python-docx. Любая попытка нарисовать их вручную проваливается.

### 5.2 Как делать правильно

```python
# 1) Скопируй шаблон ученика как стартовую точку
shutil.copy(STUDENT_TEMPLATE, OUT_PATH)
doc = Document(str(OUT_PATH))

# 2) Подмени только тексты (ФИО, тема, группа, даты, число листов).
#    Не трогай структуру XML.

# 3) Удали старый текст тела документа после «якорного» параграфа
#    (того, где <wp:anchor> с большим штампом).

# 4) Сгенерируй своё тело: СОДЕРЖАНИЕ → ВВЕДЕНИЕ → разделы → …

# 5) Сохрани, проверь по чек-листу, отдай.
```

### 5.3 Что НЕ трогать в шаблоне

- `word/header1.xml` (рамка + малый штамп) — менять только текстовые
  поля внутри, не структуру.
- `word/header2.xml` (пустой) — оставить как есть.
- `<w:sectPr>` в `document.xml` — особенно `<w:titlePg/>` и
  `<w:headerReference>` (привязка к header1/header2).
- `<wp:anchor>` с большим штампом в первом параграфе тела — менять
  только текст внутри, не структуру.
- Стили титульного листа.

</template_strategy>

## 6. Подмена текстов

<text_substitution>

### 6.1 Проблема фрагментированных runs

В DOCX текст одного абзаца часто разорван на десятки `<w:t>` элементов
(Word делит при правках). Поэтому простой `text.replace()` по XML не
работает: «Азатов Р.К.» может оказаться разбит на
`<w:t>Азатов</w:t> <w:t> Р.К.</w:t>`.

### 6.2 Правильный способ

```python
def _do_substitution_in_runs(paragraph_xml, substitutions):
    """Склеить тексты всех <w:t> внутри параграфа, применить замены,
    положить результат в первый <w:t>, остальные обнулить."""
    ts = paragraph_xml.findall('.//' + qn('w:t'))
    if not ts:
        return
    full = ''.join(t.text or '' for t in ts)
    new = full
    for src, dst in substitutions:
        new = new.replace(src, dst)
    if new == full:
        return
    ts[0].text = new
    ts[0].set(qn('xml:space'), 'preserve')
    for t in ts[1:]:
        t.text = ''
```

Этот же метод используй для подмены текста и внутри `<wp:anchor>` (там
тоже есть `<w:t>` элементы), и в `header1.xml` (через распаковку zip).

### 6.3 Что именно подменять (пример)

Сначала открой шаблон и пройдись `grep`-ом по `<w:t>`, чтобы понять,
какие строки там лежат. Типовой набор замен:

| Шаблон ученика (что есть)                 | На что заменить                       |
|-------------------------------------------|---------------------------------------|
| `«QI Инжиниринг»` (тема)                  | `«Базы данных»` (или новая тема)      |
| `Азатов Радмир Кахарманович` (титул)      | ФИО студента полностью                |
| `Азатов Р.К.` (большой штамп — Выполнил)  | Инициалы студента                     |
| `Аубакиров Еркебулан Еркинович` (титул)   | ФИО руководителя полностью            |
| `Аубакиров Е.Е.` (большой штамп — Принял) | Инициалы руководителя                 |
| `Группа: РПО3 21-Р`                       | Новая группа                          |
| `Курс: 4`                                 | Новый курс                            |
| `«09» октября 2024`                       | Новая дата начала практики            |
| `«24» октября 2024`                       | Новая дата конца практики             |
| `У25` (Литера «У» + Листов «25»)          | `У34` (где 34 — фактическое число листов) |

<substitution_warning>
- Применяй замены **от длинных к коротким**, иначе «Аубакиров» съест
  замену «Аубакиров Еркебулан Еркинович».
- Лучше — полные строки целиком.
- После генерации тела пересчитай **фактическое число страниц** и
  второй раз подмени «У{N}» → «У{actual_pages}».
</substitution_warning>

### 6.4 Где искать тексты

- **Титульная страница**: в `word/document.xml`, в первых ~3 KB до
  первой `<w:sectPr>`.
- **Большой штамп**: внутри `<wp:anchor>` в первом параграфе секции
  тела (после второй `<w:sectPr>`).
- **Малый штамп**: в `word/header1.xml`. Обычно там только литералы
  («РК ИТК 4S06130103 УП Отчет», «Изм», «Лист», «№ докум», «Подпись»,
  «Дата») — менять чаще всего не надо.

### 6.5 Подмена в header1.xml

python-docx не открывает напрямую содержимое сложных колонтитулов с
группами фигур. Поэтому header1.xml правится через `zipfile`:

```python
import zipfile, shutil, re

shutil.copy(STUDENT_TEMPLATE, OUT_PATH)
with zipfile.ZipFile(OUT_PATH, 'r') as z:
    header_xml = z.read('word/header1.xml').decode('utf-8')

# применить замены (как в _do_substitution_in_runs, но к строке)
for src, dst in HEADER_SUBSTITUTIONS:
    header_xml = header_xml.replace(src, dst)

# перепаковать
_replace_in_zip(OUT_PATH, 'word/header1.xml', header_xml.encode('utf-8'))
```

## 7. Перестройка тела документа

<body_rebuild>

После подмены текстов нужно убрать чужой контент и вставить свой.

```python
body = doc.element.body

# 1) Найти параграф с <wp:anchor> (первый параграф секции тела)
anchor_ns = '{http://schemas.openxmlformats.org/drawingml/2006/wordprocessingDrawing}anchor'
anchor_para = next(
    p for p in body.iter(qn('w:p'))
    if p.find('.//' + anchor_ns) is not None
)

# 2) Найти ВСЕ <w:sectPr>. Часть из них вложены в параграфы (continuous
#    section breaks), часть — прямые дети body. Финальная — последняя
#    среди прямых детей.
all_sectprs = list(body.iter(qn('w:sectPr')))
final_sect = [c for c in body if c.tag == qn('w:sectPr')][-1]

# 3) Удалить ВСЁ между anchor_para и final_sect
children = list(body)
i_anchor = children.index(anchor_para)
i_final  = children.index(final_sect)
for el in children[i_anchor + 1 : i_final]:
    body.remove(el)
```

После этого вставляй новые абзацы перед `final_sect`:

```python
final_sect.addprevious(my_paragraph_xml)
```

### 7.1 Что НЕ трогать в body

- `header1.xml`, `header2.xml`.
- `<wp:anchor>` в первом параграфе.
- Все `<w:sectPr>` блоки.
- `<w:titlePg/>` и `<w:headerReference>`.

### 7.2 Поправка полей в финальной секции

При необходимости поправь `<w:pgMar>` в `final_sect` на 30/10/15/30:

```python
pgmar = final_sect.find(qn('w:pgMar'))
pgmar.set(qn('w:left'),   str(int(30 * 567 / 10)))   # mm → twips
pgmar.set(qn('w:right'),  str(int(10 * 567 / 10)))
pgmar.set(qn('w:top'),    str(int(15 * 567 / 10)))
pgmar.set(qn('w:bottom'), str(int(30 * 567 / 10)))
```

</body_rebuild>

## 9. Рисунки и таблицы

<figures_and_tables>

- Нумерация **в пределах раздела**: «Рисунок-1.1», «Рисунок-1.2», …
  «Рисунок-2.1». **Не сквозная.**
- Подпись рисунка — **под** картинкой, по центру, TNR 12 курсив.
- Подпись таблицы — **над** таблицей, с абзацного отступа 1.25, TNR 12.
- В тексте обязательно ссылайся: `(рисунок 1.2)`, `(таблица 2.1)`.
- Подпись должна быть содержательной: не «Скриншот», а «Дашборд Sample
  Database в Metabase: агрегация продаж по категориям».

### 9.1 Скриншоты программ

<screenshot_rules>
- В кадре должна быть **только программа**: никакого рабочего стола,
  панелей задач, иконок ОС.
- Для браузерных приложений — либо весь браузер целиком (без панели
  задач), либо только полезная область программы внутри браузера.
- Power BI Desktop / Tableau Desktop нативно под Linux не запускаются.
  Варианты:
  - Power BI Service / Tableau Public в браузере (требуют учётной
    записи).
  - Локально Metabase или Apache Superset — open-source BI, ставятся
    одной командой, дают «скрин только программы».
- Размер картинок ≤ 15 см по ширине, чтобы помещались внутрь рамки.
</screenshot_rules>

</figures_and_tables>

## 10. Источники, ссылки, цитирование

<bibliography>
- **Минимум 15 источников.**
- Ссылки в тексте — в квадратных скобках: `[1]`, `[7]`.
- Формат печатного источника:
  ```
  Фамилия И.О. Название. — Город: Издательство, год. — N с.
  ```
- Формат электронного:
  ```
  Документация X [Электронный ресурс]. — Режим доступа: URL.
  — Дата обращения: ДД.ММ.ГГГГ.
  ```
- Список выравнивается по ширине, абзацный отступ 1.25 см.
- Нумерация арабская, сквозная.
</bibliography>

## 11. Стиль изложения

## 12. Проверка перед выдачей

<validation_checklist>

### 12.1 Базовая валидация

1. Файл открывается без ошибок:
   ```python
   from docx import Document
   doc = Document(str(OUT_PATH))   # не должно падать
   ```
2. Структура: титул → СОДЕРЖАНИЕ → ВВЕДЕНИЕ → 1 ОСНОВНАЯ ЧАСТЬ →
   2 СПЕЦИАЛЬНАЯ ЧАСТЬ → ЗАКЛЮЧЕНИЕ → СПИСОК → (ПРИЛОЖЕНИЯ).
3. Шрифт: TNR 14 в основном тексте, TNR 12 в подписях.
4. Жирного **нет** нигде, кроме унаследованного от стилей титула.
5. Поля 30/10/15/30 мм. Проверяй через `<w:pgMar>` в последней
   `<w:sectPr>`.
6. ≥ 27 страниц. Проверяй:
   ```bash
   pdfinfo file.pdf | grep Pages
   ```
7. Подпись «Рисунок-N.M …» под каждым изображением; ссылки на рисунки
   и таблицы в тексте есть.
8. СОДЕРЖАНИЕ соответствует фактическим номерам страниц.
9. Источников ≥ 15.
10. На странице СОДЕРЖАНИЕ — большой штамп; на остальных страницах тела
    — рамка + малый штамп. На титуле — ничего.

### 12.2 Проверка целостности шаблона

`libreoffice --headless --convert-to pdf …` **не отрисует** большой
штамп и часто не отрисует рамку в режиме `<wp:anchor>` со сложной
VML-группой. Это **нормально**: студенческий шаблон сам по себе так же
выглядит в LibreOffice. В Word рамка и штампы появятся.

Чтобы убедиться, что структура цела, сравни XML с шаблоном-эталоном:

```bash
diff <(xmllint --format student_unzip/word/header1.xml) \
     <(xmllint --format my_unzip/word/header1.xml)
```

Если `diff` пуст — `header1.xml` скопирован байт-в-байт.

Внутри `<wp:anchor>` в `document.xml` сравни количество критичных
элементов:

```python
for el in ('wp:anchor','a:graphic','w:tbl','mc:Fallback','v:group','v:rect'):
    assert s_anchor.count('<'+el) == m_anchor.count('<'+el)
```

</validation_checklist>

## 13. Антишаблоны (что НЕ делать)

<anti_patterns>

1. ❌ **Не** строй штампы (`build_small_stamp`, `build_large_stamp`)
   из таблиц `python-docx`. Они никогда не выглядят как в шаблоне.
2. ❌ **Не** добавляй `<w:pgBorders>` к sectPr ради рамки — настоящая
   рамка живёт в `header1.xml` как часть VML/DrawingML группы.
3. ❌ **Не** меняй структуру `<w:sectPr>` в шаблоне (особенно
   `<w:titlePg/>` и `<w:headerReference>`). Это сломает чередование
   первой и последующих страниц.
4. ❌ **Не** удаляй `<wp:anchor>` из первого параграфа тела — без него
   страница СОДЕРЖАНИЕ останется без большого штампа.
5. ❌ **Не** ставь жирный (`<w:b/>`) в теле. Регламент явно запрещает.
6. ❌ **Не** делай сквозную нумерацию рисунков «Рисунок 1, Рисунок 2…».
   Только «Рисунок-1.1», «Рисунок-1.2».
7. ❌ **Не** ставь точку после номера подраздела («1.1.»).
8. ❌ **Не** меняй стили титульного листа (жирность, размер, центр) —
   это фиксированный блок шаблона.
9. ❌ **Не** запускай `python`/`pip` в системном окружении. Только venv.
10. ❌ **Не** клади служебные файлы (планы, todo, черновые скрипты,
    промежуточные скриншоты) в финальную выдачу пользователю.
11. ❌ **Не** генерируй отчёт без шаблона ученика. Стоп → попроси шаблон.
12. ❌ **Не** забывай пересчитать число страниц после генерации и
    подменить `У{N}` в большом штампе на актуальное значение.

</anti_patterns>

## 14. Высокоуровневый pipeline

<pipeline>

```python
def main():
    # 1. Скопировать шаблон ученика как основу
    shutil.copy(STUDENT_TEMPLATE, OUT_PATH)
    doc = Document(str(OUT_PATH))

    # 2. Подменить тексты на title page + в большом штампе
    for p in doc.element.body.iter(qn('w:p')):
        _do_substitution_in_runs(p, SUBSTITUTIONS)

    # 3. Найти якорный параграф (<wp:anchor>) и финальный <w:sectPr>
    body = doc.element.body
    anchor_para = _find_anchor_para(body)
    final_sect  = _find_final_sect(body)

    # 4. Удалить всё между ними
    _strip_old_body(body, anchor_para, final_sect)

    # 5. Поправить pgMar финальной секции на 30/10/15/30 (если нужно)
    _set_margins(final_sect)

    # 6. Построить новое тело перед final_sect:
    heading_struct('СОДЕРЖАНИЕ');     render_toc(toc_entries)
    heading_struct('ВВЕДЕНИЕ');       render_items(C.INTRO, '1')
    heading_section('1 ОСНОВНАЯ ЧАСТЬ')
    for n, sub in main_part_subsections:
        heading_subsection(f'1.{n} {sub.title}')
        render_items(sub.items, '1')
    heading_section('2 СПЕЦИАЛЬНАЯ ЧАСТЬ')
    for n, sub in special_part_subsections:
        heading_subsection(f'2.{n} {sub.title}')
        render_items(sub.items, '2')
    heading_struct('ЗАКЛЮЧЕНИЕ');     render_items(C.CONCLUSION, '2')
    heading_struct('СПИСОК ИСПОЛЬЗОВАННОЙ ЛИТЕРАТУРЫ')
    for i, src in enumerate(C.BIBLIO, 1):
        add_body_paragraph(body, f'{i}. {src}', before_el=final_sect)

    # 7. Сохранить
    doc.save(OUT_PATH)

    # 8. Пересчитать число страниц и подменить «У{N}» в большом штампе
    pages = _count_pages_pdf(OUT_PATH)
    _patch_pages_in_header_and_anchor(OUT_PATH, pages)

    # 9. Сконвертировать в PDF для предпросмотра
    subprocess.run([
        'libreoffice', '--headless', '--convert-to', 'pdf', str(OUT_PATH)
    ])
```

Готовый скелет — см. `assets/build_from_student.py`.

</pipeline>

## 16. Что выдавать пользователю

<delivery>

В финальном сообщении:

1. **Краткое описание** содержимого: страниц, разделов, источников,
   тема, ФИО студента.
2. **Два вложения**: `.docx` и `.pdf`. Имена файлов — латиницей
   (платформа режет/портит кириллицу), пользователь переименует сам.
   Пример: `Balykbaev_report_BI.docx`, `Balykbaev_report_BI.pdf`.
3. **Предупреждение**: если открывает PDF в просмотрщике, в котором не
   видно рамки или большого штампа — пусть откроет DOCX в Microsoft
   Word, там рамка и штампы рендерятся корректно. LibreOffice не
   отрисовывает сгруппированные VML-фигуры.
4. **Не** клади в выдачу служебные файлы (мои скрипты, промежуточные
   скриншоты, todo-листы, планы).

</delivery>
